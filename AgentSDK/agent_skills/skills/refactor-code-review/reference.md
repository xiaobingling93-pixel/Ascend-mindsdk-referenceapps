# 架构与重构分析参考

## 架构分析维度

### 1. 分层与模块边界
- 依赖方向是否单向（如领域不依赖基础设施）
- 模块间耦合度
- 循环依赖检测

### 2. 代码坏味道
- **过长函数/类**：>50行函数、>500行类需关注
- **重复代码**：DRY 违背
- **过度抽象/欠抽象**：不必要的间接层 vs 明显应抽离的逻辑
- **职责不清**：类/模块做了多件事

### 3. 设计模式适用性
- 现有实现是否可套用模式简化
- 过度设计 vs 合理抽象

### 4. 可测试性
- 依赖注入情况
- 纯函数 vs 副作用集中
- Mock 难度

## 常见重构建议映射

| 问题 | 建议 |
|------|------|
| 长函数 | 提取方法、拆分职责 |
| 重复代码 | 提取公共函数/基类 |
| 循环依赖 | 引入接口/事件解耦 |
| 全局状态过多 | 依赖注入、收窄作用域 |
| 条件分支复杂 | 策略/多态、表驱动 |
| 参数过多 | 参数对象、建造者 |

## PR 审查 vs 全仓审视差异

| 维度 | PR 审查 | 全仓审视 |
|------|---------|----------|
| 范围 | 变更文件 | 选定模块/全仓 |
| 重点 | 新代码质量、与现有风格一致性 | 历史债务、全局架构 |
| 粒度 | 函数/类级 | 模块/分层级 |
| 输出 | 合并前改进项 | 中长期重构路线图 |
